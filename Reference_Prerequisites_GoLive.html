Account Setup Prerequisites

The agreements, structures, and tools each account needs before the intake process is operational. Complete Phase 1 prerequisites first and validate the manual process before enabling the AI Agent. The AI Agent is Phase 2 — it does not go live until the manual pilot is complete and the team has signed off.

🚶
Manual Process First — AI Agent is Phase 2 Before the AI Agent is enabled, the full intake process must be run manually per account. The PDM performs all intake actions — form submission, Zoho ticket creation, Basecamp card management, Slack notifications, and Smartsheet portfolio logging — until the process is validated and the team is aligned. Only then does the AI Agent take over those steps.
📋
Smartsheet Forms — one set per account Each account requires a New Request form and a Reported Issue form built directly on their Client Request Portfolio sheet in Smartsheet. Both forms must be configured, tested, and linked on the Basecamp Request Board before the account goes live. The Anago account is the pilot — forms are built and ready to test.

🤝 1 — Account Structure Alignment with Client

✓
Budget attribution and allocation agreed per service epic
Client stakeholders must confirm the hour allocation per service epic — Planning/Ops/New Requests, Production Support, Maintenance/Development, Deployments/Releases, Buffer — for each billing cycle before any work is tracked against those epics.
✓
Severity level definitions agreed and documented (Sev 1–4)
What constitutes Sev 1 through Sev 4 must be defined, agreed, and documented with each client stakeholder per account — covering impact criteria, examples, and ownership of severity assignment at intake. These definitions are loaded into the AI Agent so it can reference them accurately in notifications.
✓
SLA response times defined per severity level
Initial response time, update cadence, and resolution target must be defined per severity level and agreed with client stakeholders. These are the commitments the AI Agent communicates to the requestor at intake and monitors against continuously throughout the production issue workflow.
✓
Communication channels and escalation paths confirmed
Which tools are used for what (Slack, Basecamp, email), who gets notified at each stage, and the escalation path for Sev 1 issues and budget overages — all confirmed per account and loaded into the AI Agent's notification configuration.
✓
Release versioning approach agreed per account
Hotfix vs. scheduled release protocol per severity level must be confirmed with client stakeholders. This governs how and when production fixes are deployed and aligns with any app store submission windows or client-specific deployment constraints.

🛠️ 2 — Production Support Operational Structure

✓
Dedicated production support team assigned per account
A dedicated Dev + QA resource must be assigned to each account's production support function — with clearly defined coverage windows and escalation ownership. These resources must not be double-booked against delivery or planning activities during their coverage windows.
✓
Production support scheduler configured
Office hours, triage windows, and on-call coverage defined per account and communicated to client stakeholders. Coverage windows must be confirmed before the production issue workflow goes live so the team can respond to reported issues within agreed windows.
✓
Budget overage escalation path defined
Stakeholder contacts and the approval process for production support budget overages must be identified per account before the workflow goes live. This is required for the AI Agent's overage escalation trigger to reach the correct person when the Production Support epic approaches its baseline threshold.

📅 3 — Planning & Delivery Operational Structure

✓
Planning ceremony schedule established per account
Cadence, attendees, and committed resources for all planning ceremonies must be defined and confirmed per account. This schedule governs when new requests are picked up by the analysis team and is used by the AI Agent to detect inactivity on new request tickets.
✓
Activity and resource allocation scheduling confirmed — no double-booking
All ceremonies, analysis activities, and active delivery work must be mapped across accounts to prevent double-booking of shared resources. Capacity must be confirmed before any new request is picked up — accounting for meeting load, ceremony commitments, and active delivery work simultaneously across all accounts.
✓
Definition of Done (DoD) defined and agreed with all stakeholders
What can be confidently called "complete and delivered" within Concepta's control must be defined and aligned with all stakeholders per account. The DoD governs estimation inputs, delivery timing, and the DoD checkpoint in the delivery pipeline — without it, estimation is unreliable and delivery commitments cannot be made with confidence.

🧰 4 — Delivery Toolkit

✓
Scope document template configured per account
A standardized scope document template must be in place and ready to be populated at the start of analysis for every new request. This is the deliverable presented to the client for review and approval before any development work begins.
✓
Notion internal board configured per account
A Notion workspace per account must be set up as the internal analysis board — used by the PDM and Analysis Team to capture request definition, requirements, T-shirt sizing, LOE estimates, and supporting artifacts. The PDM creates the board item during the AI Agent handoff confirmation step.
✓
Smartsheet external request board active per account
The Smartsheet external request board must be active for the account. The AI Agent logs a row for every request at intake — providing a consistent, structured data record of all requests received, independent of the card-based Basecamp view.
✓
Smartsheet Estimator template configured per account
The estimator must account for all activities required to reach the Definition of Done — scoping, development, QA, UAT, deployment, documentation, and any account-specific activities. Configured and validated per account before estimation work begins.
✓
Smartsheet Project Plan template configured per account
Initiated at the start of analysis on every new request — tracks scope, activities, timeline, resource allocation, and delivery milestones. Template must be configured per account and ready to launch at the point the PDM initiates Diagram 2 (New Request Workflow).

🤖 5 — Tooling & AI Agent Configuration

✓
Basecamp 4 upgrade and Request Board configured per account
All accounts must be on Basecamp 4 with a Request Board created using the standardized column structure: Validate & Assign · Analyzing / Scoping / Estimating · Triage / Troubleshooting · Review Scope / Approve / Prioritize · Done / Closed. Both intake forms must be built and linked directly on the board.
✓
Zoho Sprints configured — backlog buckets, epics, and budget hours
Backlog Sprint Buckets (Reported Issues · New Requests: Analysis) and Billing Epics (Production Support · Planning/Ops/New Requests) must be created with baseline budget hours associated before any tickets are generated by the AI Agent.
✓
Slack channels active with correct team and account mappings
Account Slack channels must be active with the correct team members mapped per request type. All Slack handles for PDM, SA, and Production Support team members must be loaded into the AI Agent configuration.
✓
AI Agent monitoring thresholds configured per account
Inactivity windows (planning ceremony cadence · SLA update cadence per severity), budget threshold alert levels, and SLA breach thresholds must all be defined and loaded per account before the AI Agent's continuous monitoring role activates.
✓
Notification templates configured and reviewed per account
All AI Agent notification templates must be configured and reviewed with the PDM per account before go-live: receipt confirmation · SLA acknowledgement by severity · scope ready for review · clarification request · resolution confirmation · inactivity nudge · SLA breach escalation · budget threshold flag · overage approval request.
✓
Smartsheet Client Request Portfolio configured per account
Each account requires a Client Request Portfolio sheet in Smartsheet using the standard column structure. Two intake forms must be built directly on the sheet — New Request and Reported Issue — with Request Type and Status pre-populated as hidden fields. Both forms must be tested before the account goes live. The Anago account is the pilot and is ready for testing.
✓
Smartsheet → Slack notification configured per account
A Smartsheet automation rule must be set up on each account's portfolio sheet: when a new row is added → send a Slack notification to the account's dedicated channel including Request Title, Type, Requestor, Severity/Priority, and a link to the sheet. Requires Smartsheet connected to Slack once under Account → Apps & Integrations → Slack. During the manual pilot the PDM posts the Slack message manually — this automation replaces that step in Phase 2.

⏳ Future Phase — Not Required for Phase 1

These items are tracked here for planning visibility. They do not block Phase 1 activation and should be actioned as the framework matures.

⏳
RCA trigger criteria agreed per account
Which severity levels and business impact thresholds require a Root Cause Analysis document must be agreed with client stakeholders. RCA is initiated alongside the fix — it does not block resolution or deployment. Define once production issue handling is stable and patterns are identified.
⏳
RCA documentation template configured per account
The RCA template covers: timeline of events, root cause analysis, business impact scope, contributing factors, resolution summary, and preventive actions. Configure once RCA trigger criteria are agreed and the production issue workflow has been validated through the pilot.
