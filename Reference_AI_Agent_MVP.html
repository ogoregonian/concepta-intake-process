AI Agent — MVP Definition &
Per-Account Configuration

What the AI Agent does in Phase 1, the acceptance criteria it must meet, and the per-account data and configuration it requires to operate correctly. Built and owned by Concepta's internal labs team.

← Executive Brief
⚠️
The AI Agent cannot operate without documentation and account configuration

The agent reads from explicitly defined workflows and account-specific content models. If those do not exist — or are incomplete — the agent routes incorrectly, notifies the wrong people, and fires against the wrong thresholds. Documentation and configuration are not optional prerequisites. They are the infrastructure the agent runs on. The manual process must be validated through the corporate site pilot before the AI Agent is enabled on any live account.

1
Phase Overview

What the AI Agent Does — Phase 1 vs. Phase 2

The AI Agent is being introduced as a scoped MVP. Phase 1 focuses exclusively on intake validation, ticket creation, routing, and continuous monitoring. Phase 2 will expand capabilities as the manual process is validated and stabilized.

Phase 1 — MVP (Current Scope)
What the AI Agent Does Now
  • Reads the form type selected by the requestor — New Request or Reported Issue — and reads the severity level assigned on reported issues
  • Validates required fields for the request type; notifies the requestor of missing data and holds the item in the Validate & Assign column until all gaps are resolved
  • Creates the item on the account's Basecamp 4 Request Board in the correct column based on request type
  • Assigns the item to the correct internal team — Analysis Team for new requests, Production Support Team for reported issues
  • Creates the corresponding Zoho Sprint backlog ticket in the correct bucket with the correct ticket type (Task for new requests, Bug for production issues)
  • Tags the ticket with the correct billing epic — Planning/Ops/New Requests or Production Support
  • Notifies the account Slack channel with ticket link, request summary, requestor name, confirmed category, severity (if applicable), and assigned epic
  • Sends receipt confirmation to the requestor with SLA response times per severity level
  • Runs continuous monitoring throughout the full workflow — inactivity nudges, requestor reminders, budget threshold flags, SLA breach escalation, revision routing, and resolution notifications
Phase 2 — Future (Post-MVP)
Planned Expansions
  • Assist with deeper request categorization beyond the two MVP request types
  • Surface relevant historical requests or documentation during the analysis stage
  • Identify inconsistencies or quality gaps in request submissions before they reach the team
  • Recommend implementation approaches based on precedent from similar prior requests
  • Expand monitoring triggers as the process matures and new patterns are identified

Phase 2 capabilities will only be scoped once Phase 1 is validated and stable across at least one live account.

2
MVP Acceptance Criteria

Eight Criteria the Agent Must Meet Before Go-Live

The AI Agent is considered ready for live account operation only when all eight criteria are met and validated through the corporate site pilot. Each criterion must be tested explicitly — not assumed.

1
Request category and severity correctly read from submitted data
The agent correctly reads the form type selected by the requestor and the severity level assigned on reported issues — routing to the correct workflow without any manual input from the PDM on standard submissions.
2
Field validation works for both request types
The agent correctly validates required fields for the confirmed request type — 4 fields for New Requests, 10 fields for Production Issues. Missing fields trigger a direct notification to the requestor before any ticket is created. The item remains held in the Validate & Assign column until all gaps are resolved.
3
Basecamp 4 item created in the correct column
The item lands in the correct Basecamp Request Board column — "Analyzing / Scoping / Estimating" for new requests, "Triage / Troubleshooting" for production issues. The account must have Basecamp 4 active with a configured Request Board using the standardized column structure.
4
Item assigned to the correct internal team
The Basecamp item and Zoho ticket are assigned to the correct team automatically — Analysis Team (PDM + SA) for new requests, Production Support Team (Dev + QA) for reported issues. The assigned team receives a Slack notification and can begin review immediately.
5
Correct billing epic and ticket type assigned
New requests and enhancements → Planning/Ops/New Requests epic + Task ticket type. Production issues → Production Support epic + Bug ticket type. Each epic must have the account's baseline budget hours loaded so remaining budget is visible at ticket creation.
6
Slack notification sent with complete and accurate content
The account Slack channel receives the ticket link, request summary, requestor name, confirmed request category, severity level (if applicable), and assigned billing epic — within the defined SLA acknowledgment window for that account.
7
PDM human oversight confirms before the request advances
The PDM reviews the AI Agent's ticket output for accuracy — confirmed category, epic assignment, field completeness, and correct Basecamp column — before triage or analysis begins. This is a mandatory oversight step, not an optional one.
8
All continuous monitoring triggers fire correctly
Every rule-based monitoring trigger — inactivity flags, requestor reminders, SLA breach escalation, budget threshold alerts, revision routing, and resolution notification — fires at the correct threshold and reaches the correct recipient via Slack or requestor notification. Each trigger must be tested individually during the pilot.
3
Per-Account Configuration Requirements

What the Agent Needs from Each Account to Operate

The AI Agent is account-aware — it reads from configuration data that is defined per account before go-live. Every account the agent operates on requires its own full configuration pass. Values differ per account — a single shared configuration does not exist.

📌
Joint responsibility — internal labs team and PDM

The internal labs team owns the technical configuration — Zoho and Basecamp integrations, Slack webhook setup, threshold loading, and notification template wiring. The PDM owns the account-specific values — SLA definitions, severity criteria, team assignments, escalation contacts, and stakeholder sign-off. Both must be complete and validated before the agent is enabled on any account.

Routing & Categorisation
📋 Intake form links — New Request form and Reported Issue form URLs linked directly on the account's Basecamp Request Board
🏷️ Request type mapping — confirms the two valid form types and their corresponding routing destinations for this account
⚡ Severity level definitions (Sev 1–4) — impact criteria and examples per level, agreed with client stakeholders, loaded so the agent can reference them accurately in notifications
Ticket & Epic Assignment
📋 Zoho project ID — the account's Zoho Sprint project where all tickets are created
🏷️ Backlog Sprint Bucket IDs — Reported Issues bucket · New Requests: Analysis bucket
💰 Billing Epic IDs — Production Support epic · Planning/Ops/New Requests epic — with baseline budget hours loaded against each
👤 Default assignee mappings — Analysis Team (PDM · SA) for new requests · Production Support Team (Dev · QA) for reported issues
SLA & Monitoring Thresholds
⏱️ SLA response times per severity — initial response, update cadence, and resolution target for Sev 1–4
⏱️ Planning ceremony cadence — used to detect new request ticket inactivity and trigger PDM nudge
⏱️ SLA update cadence per severity — used to detect production issue inactivity and trigger Production Support nudge
⚠️ SLA breach threshold — the point at which the agent re-escalates in Slack and flags to the PDM
💰 Budget threshold alert level — percentage of each service epic that triggers a proactive flag before overage occurs
👤 Budget overage escalation contacts — stakeholder name and Slack channel to notify when overage sign-off is required
Notifications & Slack
💬 Account Slack channel ID — where all agent handoff notifications, nudges, and escalations are posted
💬 Slack assignee handles — individual Slack handles for PDM, SA, and Production Support team members per account
📧 Requestor notification templates — receipt confirmation · SLA acknowledgement by severity · scope ready for review · clarification request · resolution confirmation · no-response reminder
📧 Internal notification templates — handoff summary · inactivity nudge · SLA breach escalation · budget threshold flag · overage approval request
Production Support Specifics
🕐 Production support office hours — days and hours when the Production Support team is actively covering this account
📞 On-call protocol — escalation path for Sev 1 issues that occur outside defined office hours
🔖 RCA trigger criteria — which severity levels and business impact thresholds require RCA initiation for this account
🚀 Release versioning approach — hotfix vs. scheduled release per severity level, confirmed with client stakeholders
Basecamp Board Configuration
🗂️ Basecamp project ID — the account's Basecamp 4 project where the Request Board lives
🗂️ Request Board column IDs — Validate & Assign · Analyzing / Scoping / Estimating · Triage / Troubleshooting · Review Scope / Approve / Prioritize · Done / Closed
🔗 Intake forms embedded on board — New Request form and Reported Issue form pinned and accessible to the requestor directly from the board — no separate link needed
