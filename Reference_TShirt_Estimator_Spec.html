T-Shirt Sizing Estimator — Smartsheet Build Spec

Column-by-column structure for the t-shirt sizing estimator. Three sheets: a reference table the team consults when sizing, a per-request sizing form where new requests get logged, and a calibration log added later once delivery data exists. Build this in the same Smartsheet workbook as the existing LOE estimator so the two tools sit next to each other.

Purpose & Honest Caveat T-shirt sizing exists to give clients a quick directional range when full LOE estimation isn't appropriate yet — usually because we don't know enough (POC needed) or the client wants a number now, not in three days. The starting hour ranges below are industry-typical patterns, not Concepta-calibrated values. The first version will be wrong; the real value emerges after 6–8 weeks of pilot data when we can compare original sizes to actuals and refine the bands. That refinement is the whole point.

Sheet 1 — Size Reference Table

Static reference. The team consults this when sizing a new request. Lives at the top of the workbook.
Size Hour Range Confidence What This Typically Means Next Step Example Requests
XS < 16h High Trivial — single small change, no real uncertainty, one role. Estimate directly. Skip POC. Field rename, copy update, config tweak, label change.
S 16–40h Medium-high Small — one role, contained scope, clear pattern we've done before. Estimate directly. Add a filter, simple form, minor report, single new endpoint.
M 40–120h Medium Medium — multiple roles, some unknowns, moderate complexity. Brief team estimate. POC if architectural unknowns. New screen with backend, multi-role report, simple integration.
L 120–300h Low-medium Large — significant scope, real uncertainty, multi-week. POC strongly recommended before LOE. New feature, integration with external system, complex flow.
XL 300–600h Low Very large — multi-week, multi-resource, meaningful unknowns. POC required before LOE commitment. New module, major integration, significant rework.
XXL 600+h Very low Project-scale — too big to size as a single request. Break into multiple requests. Re-size each. New product area, major rework, multi-quarter initiative.
TBD — — We don't know yet — blocker on info, feasibility, or scope clarity. Get info or run a spike before sizing. Anything where we genuinely can't ballpark.
Calibration mechanism — important These hour ranges are starting points pulled from industry patterns at agencies of similar size and shape. They are not Concepta-calibrated. After 6–8 weeks of pilot data — once we've sized 10–15 requests and seen actuals — we revisit this table and adjust the bands based on what our actual delivery data shows. The first refinement should happen at the 6-week retro.
One thing worth deciding before launch Concepta could start with fewer bands than the 6 shown. Many agencies operate with just 4 (S / M / L / XL). More bands feel more precise but increase the time the team spends second-guessing between adjacent sizes. Recommend starting with all 6, then consolidating any that don't get used after 6 weeks. Easier to remove than to add later.

Sheet 2 — Per-Request Sizing Form

One row per request being sized. The PDM or DM creates a row when a request needs sizing instead of full LOE.
Column Type Purpose & Behavior
Request Title Text Short descriptive name of the request being sized. Match the Basecamp / Zoho ticket title.
Sizing Date Date Date the sizing was done. Auto-populate to "today" on row creation.
Account Dropdown Pick from active accounts: SFC · Heller · Kingdom Strollers · Anago · Vatos · Darihana · Corporate (pilot). Single select.
Sized By Contact / Multi-select Who performed the sizing. Usually the DM, may include team members for technical input.
Size Assigned Dropdown Single select: XS · S · M · L · XL · XXL · TBD. Required field. This is the core sizing decision.
Hour Range (auto) Formula Auto-populated from Size Assigned via lookup against Sheet 1.
=IFERROR(INDEX({Sheet 1 — Hour Range Column}, MATCH([Size Assigned]@row, {Sheet 1 — Size Column}, 0)), "—")
Confidence (auto) Formula Auto-populated from Size Assigned via lookup against Sheet 1.
=IFERROR(INDEX({Sheet 1 — Confidence Column}, MATCH([Size Assigned]@row, {Sheet 1 — Size Column}, 0)), "—")
Next Step (auto) Formula Auto-populated from Size Assigned via lookup against Sheet 1. Tells the team what to do after sizing — estimate directly, run POC, break down, etc.
=IFERROR(INDEX({Sheet 1 — Next Step Column}, MATCH([Size Assigned]@row, {Sheet 1 — Size Column}, 0)), "—")
Sizing Rationale Long Text Free text — why this size? What are the unknowns? What would change the size up or down? This is the field that protects against "I just guessed M" sizing — the rationale forces explicit thinking. Required.
Weeks to DoD (auto) Formula Optional but recommended. Translates the hour range into a delivery window using the account's average burn/week from the LOE estimator. Gives the client a useful directional answer when they ask "when can I get this?"
=Hour Range ÷ Avg Burn/Week (account)
e.g., M (40–120h) ÷ 12h/week = 3.3–10 weeks
Status Dropdown Single select: Pending Sizing · Sized · POC In Flight · Converted to LOE · Delivered · Cancelled. Tracks lifecycle of the sized request.
Linked Zoho Ticket URL Direct link to the Zoho ticket for this request. Required if the request has been ticketed.
Linked Basecamp Request URL Direct link to the original Basecamp comms thread for this request.
Actual Hours (filled later) Number Left blank until the request is delivered. Filled in from time tracking once work is complete. This is the column that powers calibration.
Variance % (auto, calc'd later) Formula How far off the original sizing was, once actuals exist. Powers Sheet 3 calibration analysis.
=IF([Actual Hours]@row = "", "", ([Actual Hours]@row - midpoint of [Hour Range]@row) / midpoint of [Hour Range]@row)
Why no per-role breakdown Notice this sheet doesn't break down hours by BE / FE / QA / UI/UX. That's intentional. T-shirt sizing's value is speed and directional accuracy. Once you start breaking down by role, you've crossed into LOE estimation territory — at which point you should switch to the full LOE estimator. Keeping these tools meaningfully different prevents the team from defaulting to t-shirt sizing for everything.

Sheet 3 — Calibration Log ADDED LATER · NOT AT PILOT START

Tracks original sizing vs. actuals. Stand up at the 6-week mark once 10+ sized requests have been delivered.
Column Type Purpose
Size Band Reference One row per band: XS, S, M, L, XL, XXL.
Original Range (current) Text What Sheet 1 currently says for this band.
Sample Count Formula How many requests of this size have been delivered. Pulls from Sheet 2.
Avg Actual Hours Formula Average actual hours across delivered requests of this size.
Median Actual Hours Formula Median (less skewed by outliers than the average).
Suggested Refined Range Formula / Manual Based on actuals, what should the range be? Often calculated as (median × 0.7) to (median × 1.4) for normal distributions, but adjust based on what your data shows.
Decision & Date Text + Date Each refresh of Sheet 1 gets a row here documenting what changed and when. Creates institutional memory.
When to first calibrate Wait until you have at least 3 delivered requests in each band you actually use. With less than that, you're calibrating on noise. For most agencies that means 6–12 weeks of pilot operation. The first calibration session is a 30-minute working meeting — pull the data, review the variance, decide what to adjust.
