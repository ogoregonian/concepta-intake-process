Risks
Requestors may inflate severity — particularly Sev 1 and Sev 2 — to trigger faster responses. The verification gate must be enforced consistently or SLA definitions erode over time.
If the Smartsheet intake form has friction or access issues, intake will revert to Slack and email immediately. Pilot one account carefully before scaling.
AI Agent miscategorization at volume will overwhelm the human verification gate. Error rate must be validated in the pilot before the agent is enabled on additional accounts.
The monitoring triggers — inactivity flags, budget alerts, SLA nudges — are not yet configured. Without calibrated thresholds they will create noise, not signal.
If the PDM becomes the default submitter across multiple accounts simultaneously, intake becomes a bottleneck at the PDM level. Monitor submission volume per account during the pilot.
No process works without team commitment. If the team does not adhere to the intake standards — logging time, updating tickets, maintaining the project plan — the entire system breaks down regardless of how well it is designed.
The Smartsheet Project Plan and Zoho activity log only have value if they are maintained in real time. Retroactive logging produces inaccurate data that cannot be used for budget tracking, capacity planning, or client reporting — and directly impacts the organisation's bottom line.
Team buy-in must be established before go-live — not assumed. The process should be walked through with the full team, questions addressed, and ownership confirmed per role before the first account is piloted.
The Internal Analysis Request doc is only as useful as the depth the PDM puts into it. Underdocumenting a complex request will produce an inaccurate estimate and create misalignment during delivery. The PDM must use judgment on documentation depth — this is a behavior dependency, not a tool dependency.
Status
This is a pilot-ready process — not a production-ready system. Manual validation on one account must be completed before the AI Agent is enabled.
The AI Agent should be enabled in phases — intake routing first, notifications second, monitoring third.
Assumptions
Severity definitions (Sev 1–4) are agreed and documented with the client before go-live.
The client is actively using Basecamp as their request tracking board.
All client accounts will be upgraded to Basecamp 4 before go-live — this is required to support the Request Board setup and card-based workflow.
Smartsheet intake forms are accessible to the requestor without login barriers or friction.
The AI Agent reads form values as submitted — it does not infer or interpret.
All SLA response times, communication channels, and escalation paths are defined per account before the agent is enabled.
Budget allocations per service bucket are established and baselined in Zoho before intake begins.
The Delivery Manager actively maintains the Smartsheet Project Plan in real time — not retroactively.
For requests estimated at 10 hours or under, formal scoping documentation is at the team's discretion — a one-line description is acceptable. Estimated effort must always be recorded and the item must be added to the Release Project Plan and prioritized. Traceability is non-negotiable regardless of request size.
For lean or time-poor client accounts, the PDM is the default submitter. The client provides the context — the PDM captures and submits it. This is agreed per account before go-live.
Considerations
Slack links and Zoho URLs in this document link to reference accounts only — both are configured per account.
Version history on each card is locked at v1.0 until the process is baselined and signed off by the team.
This process requires team buy-in before go-live. The documentation reflects the intended design — implementation sequencing to be agreed.
Future State — AI Agent: Internal Analysis Doc. The goal is for the AI Agent to initiate and pre-populate the Internal Analysis Request doc as a starting point for the PDM and Analysis Team — not build from scratch. This includes hooking into the latest production repos to sanity check feasibility, surface dependencies, flag technical considerations, and identify conflicts with existing live code before analysis begins. This is a Phase 2+ capability — the manual process must be validated first.
🚶
Phase 1 — Manual Pilot First. AI Agent is Phase 2.
Before the AI Agent is introduced, the team will manually validate the full intake process and supporting toolkit. This means:

· Every step owned by "PDM (manual pilot) → AI Agent (Phase 2)" is performed by the PDM manually during Phase 1.
· The intake forms, Smartsheet portfolio, Basecamp board, Zoho ticketing, and Slack notifications are all tested and confirmed working per account.
· The Templates and reference documents are used by the team on real requests to confirm they are accurate, complete, and practical before being locked in.

The AI Agent does not go live until the manual process is working consistently, the team is aligned, and any gaps or adjustments have been addressed. The goal is a proven process first — then automation.
