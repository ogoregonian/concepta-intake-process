Phase 1 Workflow &
Delivery Toolkit

The manual operating model for processing client requests — the structure, roles, and tooling we operate to before scaling any of this with an AI Agent. Phase 1 is the discipline pilot; Phase 2 layers automation on top of a process the team has already proven.

← Executive Brief
1
Why This Phase 1 Exists

Train Ourselves Before We Train the AI

The intake initiative includes a future AI Agent that will handle validation, routing, monitoring, and notification work. But automation can only amplify a process that already works — it cannot fix one that doesn't. Phase 1 is the team operating the workflow manually so we validate, refine, and prove the structure on real requests before any of it gets handed to an agent.

🎯
The principle

If we cannot run this workflow consistently with humans, we have no business automating it. The discipline we build in Phase 1 — clear roles, ticket hygiene, project plan rigor, budget visibility — becomes the rulebook the AI Agent will eventually operate from. Skipping the manual pilot would automate ambiguity, which is worse than no automation at all.

This page is the operating reference for Phase 1. It defines the role split, the gating signal that starts every request, the end-to-end workflow, the stuck-state playbook, the rule for keeping Zoho and Smartsheet aligned, and the literal mechanics for executing each DM responsibility. It is the artifact the team operates from during the corporate site pilot and the per-account rollout that follows.

3
The Gating Signal — Bucket Hours Per Cycle

Phase 1 Starts With Visibility

Before any request can be processed, the PDM team needs to know how much capacity exists in the relevant service buckets for the current billing cycle. The DM owns and manages this signal — and it is what enables the PDM team to start processing requests with confidence about what can be absorbed.

What the DM Manages and Communicates

Per account, per billing cycle, the DM tracks and communicates remaining hours in:

  • Planning / Ops / New Requests bucket — covers standing team operations: dailies, planning meetings, retros, client status calls. Remaining hours after operations fund analyzing, scoping, and estimating new requests
  • Production Support bucket — reported issues pull from here (triage, troubleshooting, resolution)
  • Implementation bucket — approved work pulls from here once requirements, estimate, and delivery timing are signed off; POCs from t-shirt sized requests also pull from this bucket. At approval, hours are carved out of this bucket and assigned to a request-specific epic (see Stage 8) — so "remaining Implementation hours" is what's left after all approved requests have been allocated

Monitor and recalibrate. These budgets should be tracked continuously and recalibrated as patterns emerge — that data opens conversations with the client ("we need more hours for production support" or "we need more hours to perform X"). Bucket pressure isn't just a tracking signal; it's the evidence that drives the next contract conversation.

⏳
Available hours ≠ schedulable in a single day

Budgets are low and the team is not full-time on most accounts. Even when bucket hours show capacity, the resource may not be schedulable to handle everything at once. The DM accounts for resource availability — not just budget availability — when allocating time to a request.

📅
Cadence — at minimum weekly, no dedicated forum required

Tracking is continuous via Zoho epics as time logs in. Communication of remaining hours happens at minimum once per week — during dailies, internal status calls, or as a post in the relevant account Slack channel. The point is that everyone has a pulse on what's remaining without making it a ceremony. Refresh more often as buckets approach or pass key thresholds.

The End-to-End Workflow

From Client Request to Delivery Approval

Eight stages from request capture through client approval and release train assignment. The primary path covers New Requests; Reported Issue divergences are called out inline at the stages where the flow differs. The full workflow happens against the bucket capacity managed by the DM in Section 3.

💬
Anago — current exception

Anago is handled outside the standard flow during their account transition. The workflow below applies to all other active accounts.

1

Request Captured in Basecamp

Requests come in via the channel each client already uses — most accounts use the Basecamp message board, one uses a request board. The PDM works within whichever format the account uses; what matters is that the request is captured and that subsequent refinement happens in Basecamp where it is traceable.

Client / RequestorPDM (receives)
2

Zoho Ticket Created & Assigned to DM

Once the PDM picks up the request, they create the Zoho ticket on the New Request backlog category with the following:

  • Item Type — Task for new requests, Bug for reported issues (Zoho dropdown — Story, Task, or Bug)
  • Item Name — name of the request
  • Description — link to the Basecamp comms
  • Epic — Planning / Ops / New Requests for new requests; Production Support for reported issues
  • Priority — set by the PDM; for reported issues this is how the team understands urgency (Phase 1 does not use formal SLA response targets — the PDM's priority call drives team response)
  • Request Analysis Status — custom field on the ticket template; PDM sets to New Request at creation. The current ticket assignee updates Status as their phase completes — Status follows ownership, not parked with one role (see Section 6)
  • Required participants — who the PDM needs to process this request, and whether a planning call is recommended (vs. async review) — captured in the description
  • Assign Users — the DM

The PDM then tags the DM in the ticket comments and, when needed, in the account Slack channel. This tag is what triggers the DM to begin tracking.

4

Scoping & Internal Alignment

The PDM works the request to a clear scope — directly with the client in Basecamp, and internally with whichever team members are needed. Decision point: if the request is simple enough, individual team members review and align async; if complex, the DM schedules a planning call with the team members the PDM identified on the Zoho ticket.

The output of this stage is a scope posted in Basecamp covering the high-level requirements specific to what's applicable for the request — typically FE, BE, QA, UI/UX. The client approves the scope in Basecamp.

PDM (drives)SA / FE / BE / QA / UI-UX (input)DM (schedules call if needed)Client (approves)
📋
Project plan — DM discretion based on complexity

For simple requests (e.g. change a field), no separate project plan is needed. For complex requests with multiple resources and rolling days of work, the DM initiates a Smartsheet project plan to track triage/troubleshoot or scoping/analysis activities — typically once the PDM has client-aligned requirements and there's something concrete to plan against. The DM judges per request.

⚡
Fast path — PDM discretion for simple, clear requests

When the PDM is confident the request is simple and clear, they can skip the separate requirements-approval round trip with the client. The PDM posts a lightweight confirmation in Basecamp ("just to make sure I'm interpreting correctly, you want X — the DM will follow up shortly with estimate and timing") so the requirement is captured in writing, then signals the DM to proceed straight to estimation. Internal alignment for estimation still happens — only the client-facing requirements approval loop is skipped. The DM's response in Stage 7 (requirements restated + estimate + timing + release train) becomes the single client approval moment instead of two.

Notify DM Scope is Ready for Estimation

Once the scope is approved by the client in Basecamp, the PDM updates the Zoho ticket and tags the DM in the comments and the account Slack channel. This trigger moves the request from scoping into estimation.

PDM
6

DM Generates Request Estimator & Aligns LOE

The DM generates the Smartsheet request estimator and includes the high-level requirements from the approved scope, plus all the activities required to reach Definition of Done (the DoD activities are largely reusable across requests). The team estimates async if maturity allows, or convenes a brief estimation call if needed.

Each role estimates into their column (BE Est, FE Est, QA Est, UI/UX Est). Once dev estimates land, the DM populates the hedge columns (Hedge, SA, PDM, DM, Automation Testing, Team), aligns the final LOE — including DoD activities — with the team and reviews with leadership where applicable (the estimator's Internal Approved Range by Leadership column). The DM then calculates delivery timing based on hours remaining in the implementation bucket.

♻️
Shared DoD activities — when another request is already in flight on the same release

If another request on the same release train has already accounted for shared DoD activities (client demo, e2e regression on our end), those don't need to be re-estimated for the new request. In that case the LOE for the new request includes only implementation, dev testing, code review, and QA testing — the client demo and e2e regression get folded into the work already accounted for on the in-flight request. The DM should validate this assumption with the team before applying it.

Delivery ManagerFE / BE / QA / UI-UX / SALeadership (review, where applicable)

T-Shirt Sizing Variant

If the team needs to t-shirt size the request rather than fully estimate, they use the t-shirt size estimator and align async or in a brief call. T-shirt sizing typically applies when:

  • The team doesn't yet know enough to estimate precisely — a POC is needed first
  • The client is pressing for a quick high-level estimate and doesn't want to wait for full LOE

POC handling: POCs are owned by the BE or FE resources (or both, depending on the request). POC effort pulls from the Implementation bucket. The POC validates feasibility and architectural approach — and frequently delivers a "buy one get one" benefit: once the POC is in, the team usually estimates only what remains to polish and reach DoD rather than estimating the full build from scratch.

7

DM Communicates Estimate, Timing & Release Train to Client

The DM responds to the request in Basecamp with:

  • The effort estimate (or t-shirt size, if applicable)
  • The timing to reach DoD
  • The proposed release train placement
  • If the fast path was taken in Stage 4: the DM also restates the requirements in this reply — the requirements + estimate + timing + release train message becomes the single client approval moment

Release train logic — DM works with PDM on priority before client comms:

  • The PDM knows current release train priority from the client side and is the partner for this call
  • If the request is urgent and there is a release train already in flight but still in development, it makes sense to add the request to that train
  • If the request is heavy and would delay the current release train, it is prioritized for the next release train (v#.#.#)
  • Once DM and PDM are aligned, the placement is communicated to the client

T-shirt sizing caveat: when the response is a t-shirt size, the DM also informs the client of the next step — either an estimated POC that will then inform the remaining LOE to DoD, or (if no POC is needed) the team estimates and the DM follows up with timing for DoD.

Delivery ManagerPDM
8

Client Approval & Epic Carve-Out

Client approves the estimate, timing, and release train in Basecamp. The DM updates the Zoho ticket and Smartsheet to Scope Approved. For new feature/function requests, the DM then carves the approved hours out of the Implementation bucket and stands up a request-specific epic in Zoho. Reported issues do not get carved out — all work stays under the Production Support epic.

The carve-out, step by step:

  1. Close the original request ticket under Planning / Ops / New Requests. Processing is complete; we don't want any further time logging against this ticket because it would bill implementation work to the wrong bucket.
  2. Create a new epic in Zoho titled with the request name (e.g. Orders by Resort Report).
  3. Allocate hours from the Implementation bucket to the new epic matching the estimator's TOTAL PROJECT HOURS for the request.
  4. Cut child tickets under the new epic — one per role, each carrying that role's estimated hours from the estimator: BE, FE, QA, UI/UX, SA, PDM, DM, Automation Testing, Team, and a separate Hedge ticket.
  5. The Hedge ticket is the request's internal buffer. If any role ticket runs over its estimate mid-flight, the DM pulls from Hedge first before going back to the client. Hedge is request-specific — it doesn't pool across requests, and unused Hedge stays with this request.

From there, execution moves through the standard DoD pipeline (In Dev → DoD Dev Done → Released to PROD). Smartsheet status moves to In Dev and tracks delivery progression alongside Zoho's active board.

🎯
Why we carve out at approval — budget integrity and KPI tracking

The carve-out is what keeps budgets honest. If implementation work continued logging against Planning / Ops / New Requests, that bucket would look over-burned and Implementation would look under-burned — and per-request estimated-vs-actual data would be impossible to extract. The request-specific epic gives us clean tracking at the request level: what we estimated for DoD vs. what it actually took. That's the dataset we use to refine future estimates, manage budgets in real time, and report on delivery KPIs accurately.

If the client does not approve, see Section 5 — Stuck-State Playbook.

Stuck-State Playbook

What to Do When the Flow Stalls

The workflow assumes a clean path through. In practice requests stall at predictable points — and the response depends on which condition triggered the stall and which team owns the action.

Stuck Condition Action Owner
Client doesn't approve the timing DM informs the client we can increase velocity and budget for this specific request to meet their expectations — adjusts the estimate range and release train accordingly. Delivery Manager
Client doesn't approve the estimate / cost PDM works with the client to pare down scope. Once revised scope is approved in Basecamp, the request loops back to Stage 6 for re-estimation. PDM
Estimate exceeds remaining bucket hours for the cycle DM and PDM align on options: defer to next cycle, pull from Buffer (if available and appropriate), reduce scope, or escalate for overage approval per the account's escalation path. DM + PDM
Project plan is running over mid-flight DM flags the overage early. Team and PDM review whether to absorb, reduce remaining scope, or escalate to client. The point is to flag before the date slips — not after. Delivery Manager
Client doesn't respond to scope or estimate PDM follows up in Basecamp at a reasonable cadence. If silence persists, capture the block reason as a comment on the Zoho ticket and note it on the Smartsheet row. The Request Analysis Status reflects the holding state — we don't add a Blocked status because the underlying processing state hasn't changed; it's just paused. PDM
Reported issue priority unclear PDM assigns initial priority on ticket creation. If triage uncovers different priority than initially set, the PDM updates it on the Zoho ticket and notifies the team. PDM
⚠️
Stalls are not silent

Every stuck state requires either a client communication or an internal escalation — not absorption. The discipline of surfacing stalls early is part of what we are training ourselves on in Phase 1.

Source of Truth — Three Views, One Reality

Smartsheet, Zoho Status Field, Zoho Active Board

Every request shows up in three places, each owned by a different role and serving a different audience. The trick is that they all reflect the same underlying state — but each view is built for the role that needs it most.

Smartsheet
Client View
  • Owner: DM
  • Account Request Tracker (kanban)
  • Where status is shared externally with the client
  • Project plan + estimator live here too
Zoho — Status Field
Front-Half Internal View
  • Owner: Current ticket assignee
  • Custom Request Analysis Status field on the ticket
  • Tracks processing state from intake → scope approved
  • Status follows ownership of the current phase
Zoho — Active Board
Dev Lifecycle View
  • Owner: Dev team
  • Existing columns (TO DO → IN DEVELOPMENT → ... → UAT APPROVED)
  • Takes over once Scope Approved → Sprint Candidates
  • Smartsheet "In Dev / Released to PROD" mirrors this
🧠
The Smartsheet tracker is institutional memory

Clients constantly reprioritize — what's critical today gets superseded tomorrow, and weeks later they ask "what about that other thing we discussed?" because they've forgotten. The Smartsheet tracker is how we hold the line on what was requested, when, and at what status. Without it, requests fall through the cracks and we end up in reactive whack-a-mole. The discipline of moving every request through the tracker — even when the client has gone quiet — is what protects both us and them.

The Front-Half State Set — Six Values

The Smartsheet column and the Zoho Request Analysis Status field share the same six values during the front half. The current ticket assignee updates the Status as their phase completes — PDM during intake and requirement refinement, DM during estimation, whoever is doing the work owns the move. The DM mirrors each move on the Smartsheet column for client visibility.

New Request → In Analysis / Triage → Pending Client Approval (Requirements) → Ready for Estimation → Pending Client Approval (Scope & Estimate) → Scope Approved

What happens at Scope Approved: the Zoho ticket flows from the backlog into Sprint Candidates — and from there into the active board (TO DO → IN DEVELOPMENT → ...) when capacity is assigned. The Request Analysis Status field stops being the source of truth at that point; the active-board column takes over. On Smartsheet, the row moves into In Dev and progresses through to Released to PROD mirroring the Zoho active-board state.

Blocked is a comment, not a state. If a request is blocked at any point — client silence, missing input, dependency on another team — capture it as a comment on the Zoho ticket and note it on the Smartsheet row. We don't add a Blocked status to the front-half set because the underlying processing state hasn't changed; it's just paused.

Project Plan — Distributed Update Model

When a project plan is created (DM discretion based on complexity — see Stage 4), it is the activity tracker for the request lifecycle, structured by phase (Technical Analysis → Scope & Estimate → Implementation → Dev Testing → Deployment → DoD). Updates are distributed:

  • DM owns: creating the plan from the template, structuring phases, assigning activities, sequencing, monitoring overall progress and budget burn, reconciling status
  • Activity assignees update: their own status on the activities they own — SA, PDM, FE, BE, QA, UI/UX, named stakeholders, the DM themselves
  • Cadence: as activities progress — the project plan is the live record of the request, not a weekly snapshot
📊
Why this discipline matters

Project plan rigor is what gives the DM control over resource allocation, sequencing, and budget burn — and gives the PDM and the broader team visibility into where each request actually stands. It is also the dataset the AI Agent will eventually read from. Manual discipline now is what makes future automation possible.

5 Decide on a Project Plan (DM Discretion)

What: Judge whether the request warrants its own Smartsheet project plan based on complexity. Simple requests don't; complex requests do.

  1. Skip the project plan if the request is simple — single field change, copy update, configuration tweak. The Zoho ticket and tracker row are enough.
  2. Create a project plan if the request involves multiple resources, rolling days of work, or sequencing risk. Typically initiated once the PDM has client-aligned requirements and there's something concrete to plan against.
  3. If creating: copy the project plan template, populate the analysis tasks (or triage tasks for reported issues) with owners, durations, and start dates. Update statuses as activities progress; expect activity assignees to update their own statuses on activities they own.
  4. If a request is being added to an existing project plan for a release that's already in flight, add it there rather than creating a new plan.

6 Generate & Align the Request Estimator

What: Stand up the Smartsheet request estimator and drive it to a final approved LOE range.

  1. Once the PDM signals the scope is client-approved, copy the request estimator template.
  2. Pull the high-level requirements from the approved Basecamp scope into the estimator's description column.
  3. Populate the standard DoD activities — these are largely reusable. If the request is folding into an existing release where DoD activities are already accounted for, only estimate implementation, dev testing, code review, and QA testing.
  4. Share the estimator with the team for input. Each role estimates into their column: BE Est, FE Est, QA Est, UI/UX Est. Async if maturity allows; brief estimation call otherwise.
  5. Once dev estimates land, populate the hedge columns: Hedge, SA, PDM, DM, Automation Testing, Team.

7 Calculate Delivery Timing & Align Release Train with PDM

What: Translate LOE into a delivery date and decide which release train the request rides on.

  1. Pull the average burn/week and remaining hours in the implementation bucket from the estimator's reference columns.
  2. Calculate weeks to DoD: TOTAL PROJECT HOURS ÷ Avg Burn/Week. Adjust for known scheduling constraints (resource not full-time, holidays, parallel work in flight).
  3. Bring the timing to the PDM. The PDM knows current release train priority from the client side — together decide:
    • Add to current in-flight release if urgent and the release is still in development
    • Hold for next release (v#.#.#) if the request is heavy enough to delay the current one
  4. Once aligned with the PDM, communicate estimate, timing, and release train to the client in Basecamp.

9 Keep Smartsheet and Zoho in Sync

What: The Zoho Request Analysis Status field follows ownership — whoever the ticket is assigned to during the current phase updates the Status as their work completes.   2. Whoever moves the Status adds a comment on the ticket so the next owner knows what was done and what's needed next. The comment is what disambiguates handoffs (e.g. "@DM scope approved, ready for estimation" vs. "@PDM client wants Y revised before we can estimate").

10 Communicate with the Client in Basecamp

What: Respond on the original Basecamp request thread with estimate, timing, and release train placement.

  1. Reply on the original Basecamp thread (message board or request board, whichever the account uses).
  2. Include: effort estimate (or t-shirt size if applicable), timing to DoD, proposed release train, any caveats.
  3. If t-shirt sized: name the next step (POC effort estimate, or follow-up LOE estimate).
  4. Wait for client approval. On approval, execute Mechanic 8 (the carve-out). On non-approval, see the Stuck-State Playbook.

Zoho ticket template — custom field configuration

A custom dropdown field needs to be added to the Zoho ticket template before pilot start. This is what enables the front-half status tracking described in Section 6.

Field name: Request Analysis Status
Field type: Dropdown (single select)
Default value: New Request
Required: No (so it doesn't block tickets that have moved past the front half)

Picklist values (in order):

  1. New Request
  2. In Analysis / Triage
  3. Pending Client Approval (Requirements)
  4. Ready for Estimation
  5. Pending Client Approval (Scope & Estimate)
  6. Scope Approved

Setup tip: place the field near the top of the ticket form alongside Item Type, Epic, and Priority — it should be part of the standard ticket creation flow, not buried at the bottom. Make it editable by anyone the ticket is assigned to, since status follows ownership of the current phase.
