Overview

What the Agent Does

The AI Agent is a rule-based automation layer that sits between the client request submission and the internal team handoff. It does not make decisions or exercise judgment — it reads submitted data values, validates completeness, executes a defined set of write actions across four tools, and sends pre-configured notifications. Every action is deterministic and traceable.

⚠️
Manual process must be validated first

The AI Agent must not be enabled on any live client account until the manual intake process has been validated through the corporate site pilot. The agent reads from explicitly configured account data — if that configuration is incomplete, it routes incorrectly and notifies the wrong people.

💡
Scope boundary — Phase 1 only

This spec covers Phase 1 exclusively: intake validation, ticket creation, Smartsheet logging, Basecamp card movement, Slack notification, requestor confirmation, and lightweight continuous monitoring (budget thresholds, revision routing, resolution notification). SLA cadence monitoring and breach escalation are Phase 2 and are not in scope here.

CapabilityPhaseDescription
Field validationPhase 1Read submitted form values, check all required fields are present and non-empty for the request type
Request categorisationPhase 1Read form type from submitted data — New Request or Reported Issue — and severity level on production issues
Zoho ticket creationPhase 1Create Task (new request) or Bug (production issue) ticket with correct epic, sprint bucket, and assignee
Smartsheet row loggingPhase 1Write one row to the account's external request board, mirroring intake data

System Architecture

How It Fits Together

The agent is triggered by a Basecamp webhook fired when a new form submission lands on the Request Board. It runs as a lightweight serverless function (recommended: AWS Lambda or equivalent). All actions are outbound API calls to four external services. No database is required — all state lives in the source tools.

A
Trigger — Basecamp Webhook
Entry point

Basecamp fires a webhook POST to the agent's endpoint whenever a new card is created on the Request Board. The payload includes the card content, column it was placed in, project ID, and creator details. The agent endpoint must be registered as a webhook in each account's Basecamp project settings.

Webhook event typecard_table_card.created on the Request Board card table
Payload includesCard ID, card content (form fields as text), column ID, project ID, creator name and email, timestamp
AuthHMAC signature verification using Basecamp webhook secret — validate before processing
Response requiredReturn 200 OK within 10 seconds — run processing async if needed
B
Agent Runtime
Serverless function

Recommended deployment: AWS Lambda (Node.js 18+ or Python 3.11+) behind an API Gateway endpoint. The function is stateless — it reads the webhook payload, looks up the account configuration from a config store (environment variables or a simple config file per account), and executes the action sequence.

Recommended runtimeNode.js 18+ or Python 3.11+ — either is suitable; match what the labs team is most comfortable with
Config storagePer-account config object stored as environment variables or a JSON config file — one config per account
Execution timeoutSet to 30 seconds minimum — sequential API calls to 4 services can take 10–15s
Error loggingAll failures must be logged with account ID, request ID, failed action, and error message — CloudWatch or equivalent
Retry logicRetry failed API calls once with 2s delay before marking as failed — do not retry notification sends (prevents duplicates)
C
Continuous Monitoring — Polling Loop
Scheduled job

The monitoring triggers (budget threshold, revision detection, resolution notification) run on a scheduled polling loop — not on webhooks. Recommended: a separate scheduled Lambda function running every 15–30 minutes that checks active ticket states in Zoho and Basecamp card positions against expected states.

TriggerScheduled CloudWatch event — every 15–30 minutes during business hours
Checks per runBudget spend vs threshold · Basecamp card in revision column · Zoho ticket status = Resolved
State trackingStore last-notified timestamp per ticket in a lightweight store (DynamoDB or S3 JSON) to prevent duplicate notifications

Processing Logic

Decision Tree — New Submission

When the webhook fires, the agent runs the following steps in order. If any required step fails, the agent logs the error and sends a fallback notification to the PDM Slack channel. It does not proceed to downstream steps on a failed action.

→
Step 1 — Parse Submission
1
Extract form type
Read the form type value from the card content. The requestor selects either New Request / Enhancement or Reported Issue on the Basecamp form before submitting. This value drives all downstream routing. The agent reads it — does not infer it.
2
Extract account context
Map the Basecamp project ID from the webhook payload to the correct account config object. If no config is found for the project ID, log error and send PDM fallback alert — do not proceed.
3
Extract all field values
Parse the card content to extract all submitted field values as key-value pairs. Field names must match the standardised field labels defined in the intake form spec. Store parsed values for use in all downstream actions.
→
Step 2 — Validate Required Fields

Check that all required fields for the identified form type are present and non-empty. Required fields differ by form type:

Form TypeRequired Fields (all must be present)
New Request / Enhancement 1. Request Title · 2. Problem Being Solved or Goal Being Pursued · 3. Who Is Impacted · 4. Where Does This Occur (Platform or System)
Reported Issue 1. Title of Issue · 2. Severity Level (1–4) · 3. Overview of Issue · 4. Steps to Reproduce · 5. Expected vs Actual Behavior · 6. Impacted Components · 7. Affected Platform & Version · 8. Screenshots/Video · 9. Console Logs · 10. Number of Reports

If any required field is missing: do not create any tickets or move any cards. Send a missing-fields notification to the requestor (see notification templates). Hold the card in the Validate & Assign column. Log the incomplete submission. Wait for resubmission — the next webhook from the same card triggers the full validation loop again.

→
Step 3 — Execute Actions (all fields valid)

Once validation passes, execute all actions in this order. Steps 3a–3d should be treated as a batch — if any fails, log and send PDM fallback alert.

3a
Create Zoho ticket Zoho API
New Request: Create Task ticket · Epic: Planning/Ops/New Requests · Sprint bucket: New Requests: Analysis · Assignee: PDM/Analysis Team user ID · Title: from Request Title field · Description: all submitted field values formatted as structured text

Production Issue: Create Bug ticket · Epic: Production Support · Sprint bucket: Reported Issues · Assignee: Prod Support Team user ID · Title: from Title of Issue field · Description: all submitted field values including severity level
3b
Log row to Smartsheet external request board Smartsheet API
Add one row to the account's external request board. Populate: Request Title, Request Type, Requestor name, Submission Date (today), Status = "New", Zoho Ticket Link (from step 3a response), Basecamp Card Link. Use the Zoho ticket ID returned from step 3a — do not proceed with a null ticket ID.
3c
Move Basecamp card to correct column Basecamp API
New Request: Move card to Analyzing / Scoping / Estimating column
Production Issue: Move card to Triage / Troubleshooting column

Use the column IDs stored in the account config — do not look them up at runtime.
3d
Post Slack notification Slack API
Post to the account's Slack channel. Tag the assigned team member by their Slack user ID. Include: request title, form type, requestor name, Zoho ticket link, Basecamp card link. For production issues, include severity level. See Section 5 for exact message templates.
3e
Send requestor confirmation Basecamp API
Post a message on the Basecamp card thread confirming receipt. Include: assigned team, next steps, how to track progress on the Request Board. See Section 5 for exact message templates. This step runs after 3a–3d complete — if any of those fail, do not send the confirmation.

Continuous Monitoring

Phase 1 Monitoring Triggers

The scheduled monitoring job runs every 15–30 minutes and checks for the following conditions across all active accounts. Each trigger fires at most once per event — state tracking prevents duplicate notifications.

⚡
Budget approaching threshold Phase 1
Check: Query Zoho time logs for active epics. Sum hours logged per epic for current billing cycle. Compare against baseline hours in account config. Action: If hours logged ≥ threshold percentage, send budget alert to PDM (Slack DM). State: Store budget_alert_sent_{epic}_{cycle} flag — do not re-fire for the same epic in the same billing cycle.
⚡
Requestor scope revision submitted Phase 1
Check: Poll Basecamp for cards in the Review Scope / Approve / Prioritize column that have new comments tagged as requestor feedback (detect via comment author matching requestor name on the card). Action: Move card back to Analyzing / Scoping / Estimating column. Post Slack notification to Analysis Team with revision note. Add comment to Zoho ticket logging the revision. State: Track last processed comment timestamp per card.
⚡
Zoho ticket marked Resolved Phase 1
Check: Poll Zoho for Bug tickets in the Reported Issues bucket where status = Resolved and resolution notification has not yet been sent. Action: Send resolution notification to requestor on Basecamp card thread. Move Basecamp card to Done / Closed column. State: Store resolution_notified_{ticket_id} flag to prevent re-firing.
⏳
Ticket inactive beyond SLA update cadence Phase 2
Requires SLA response time definitions to be agreed and loaded per account. Not in Phase 1 scope.
⏳
SLA breach threshold reached Phase 2
Requires SLA response time definitions to be agreed and loaded per account. Not in Phase 1 scope.

Incomplete config = incorrect routing

If any field in the config is null or missing, the agent must abort and send a PDM fallback alert. Do not allow partial config execution.

Acceptance Test Cases

Eight Criteria — Each Must Pass Before Go-Live

These map directly to the MVP acceptance criteria in the AI Agent MVP document. Each criterion must be tested explicitly against the corporate site pilot account before the agent is enabled on any live client account.

1
Request category and severity correctly read from submitted data
Test: Submit a New Request form and a Reported Issue form (with each severity level 1–4). Verify the agent correctly identifies the form type from the data — not the column it lands in — and reads the severity value exactly as submitted. Pass criteria: Zero misroutes across 8 test submissions (1 NR + 4 PI at each severity).
2
Field validation works for both request types
Test: Submit a New Request with each of the 4 required fields missing one at a time (4 tests). Submit a Reported Issue with each of the 10 required fields missing one at a time (10 tests). Pass criteria: Every incomplete submission triggers a missing-fields notification to the requestor. No Zoho ticket, Smartsheet row, or Basecamp card move occurs on any incomplete submission.
3
Zoho tickets created with correct type, epic, bucket, and assignee
Test: Submit one complete New Request and one complete Reported Issue. Verify in Zoho: NR creates a Task ticket under Planning/Ops/New Requests epic in the New Requests: Analysis bucket assigned to the Analysis Team user. PI creates a Bug ticket under Production Support epic in the Reported Issues bucket assigned to the Prod Support user. Pass criteria: Both tickets match config exactly — wrong type, epic, bucket, or assignee = fail.
4
Smartsheet row logged correctly for every submission
Test: Submit 3 complete requests (mix of types). Verify in Smartsheet that a row was added for each with correct values: Request Title, Request Type, Requestor name, today's date, Status = "New", correct Zoho ticket link, correct Basecamp card link. Pass criteria: 3 of 3 rows present with all columns populated. Zero null values in required columns.
5
Basecamp cards moved to correct columns
Test: Submit one complete New Request and one complete Reported Issue. Verify in Basecamp that the NR card moves to Analyzing / Scoping / Estimating and the PI card moves to Triage / Troubleshooting. Pass criteria: Both cards in the correct columns within 30 seconds of submission. No cards left in Validate & Assign after valid submissions.
6
Slack notifications fire with correct content and correct @mention
Test: Submit one NR and one PI (Sev 2). Verify: both messages post to the correct account Slack channel, NR tags the Analysis Team user, PI tags the Prod Support user and includes the severity level. Verify all links in the message are valid (Zoho ticket URL opens the correct ticket, Basecamp card URL opens the correct card). Pass criteria: Both messages present with no broken links and correct @mentions.
7
Requestor confirmation sent after every valid submission
Test: Submit valid NR and PI requests. Verify a confirmation message appears on each Basecamp card thread within 60 seconds. For PI, verify the severity-matched message variant is used (Sev 1 message for Sev 1 submission, etc.). Pass criteria: Confirmation present on 4 of 4 test submissions (1 NR + 1 PI at each of Sev 1, 2, 3, 4). No confirmation fires on incomplete submissions.
8
Error handling — PDM fallback alert fires on failure
Test: Simulate a Zoho API failure (temporarily use an invalid API token). Submit a complete request. Verify: no partial actions occur (no Smartsheet row, no Basecamp move, no Slack notification), a fallback alert fires to the PDM Slack DM with the error details, the card remains in the Validate & Assign column. Pass criteria: Zero partial writes on failure. PDM alert fires with account name, request title, and error description.

Implementation Notes

Guidance for the Labs Team

→
Recommended Build Sequence
1
Set up the corporate site pilot account config
Configure the config object for the corporate site account first. All four integrations must be fully populated before any testing begins. Get the Basecamp column IDs, Zoho sprint and epic IDs, Smartsheet column IDs, and Slack channel/user IDs from the PDM.
2
Build and test the intake trigger in isolation
Build the webhook receiver and field parser first. Test that it correctly parses both form types and identifies missing fields before wiring up any write actions. Use test payloads — do not run against the live Basecamp board until parsing is confirmed correct.
3
Add write actions one at a time
Wire up Zoho first, then Smartsheet, then Basecamp, then Slack. Test each integration end-to-end before adding the next. This makes debugging significantly easier and isolates failures to a single integration.
4
Run all 8 acceptance test cases
Do not mark the agent ready for go-live until all 8 criteria pass. Work through them in order — earlier criteria are dependencies for later ones. Involve the PDM in test runs to confirm the correct data is landing in Zoho and Smartsheet.
5
Build the monitoring loop separately
The scheduled monitoring job is a separate Lambda function — build it after the intake trigger is stable and passing all tests. Do not combine them into the same function.
🔐
Credentials and secrets

Store all API tokens as environment variables in the Lambda function configuration. Never commit tokens to the repository. Rotate tokens if they are ever exposed. The Basecamp webhook secret, Zoho OAuth tokens, Smartsheet API key, and Slack bot token each require separate secure storage. Use AWS Secrets Manager or equivalent for production deployment.

📋
PDM responsibilities during build

The PDM needs to supply: all Basecamp column IDs (pull from the board URL), Zoho project ID, sprint IDs, epic IDs, and assignee user IDs, Smartsheet sheet ID and column IDs, Slack channel ID and user IDs for each assignee role. The labs team should request these as a structured checklist rather than gathering them piecemeal — use the account config object in Section 7 as the handoff template.
