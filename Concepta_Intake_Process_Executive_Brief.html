Background & Current Challenges

Why This Initiative Exists

Concepta is a custom software development agency managing multiple client accounts simultaneously on a shared-resource model. As the organization scales, historically informal processes are creating measurable operational friction.

Challenge

Requests arrive through multiple informal channels — meetings, email, Basecamp, DMs, phone calls — with no single intake entry point.

Challenge

No standardized intake form or triage process leads to excessive time spent clarifying requests before work can even be evaluated.

Challenge

Incoming requests often lack the minimum information required for proper evaluation, estimation, or routing.

Challenge

Team members support multiple accounts simultaneously, making priority, timeline, and capacity visibility difficult to maintain.

Challenge

Operational workflows are not consistently documented, preventing teams from following a shared structure when processing work.

Challenge

Requests enter delivery pipelines without being assigned to the correct billing epic, creating budget tracking and reporting gaps.

Strategic Gap

Concepta is not currently utilizing AI agents within any of its agency software lifecycle operational processes — and the industry is moving fast. Agencies that integrate AI into their delivery operations now will have a compounding advantage in speed, consistency, and scalability. This initiative is the right starting point: the intake process is rule-based, repetitive, and high-volume — making it an ideal first use case to introduce an AI agent into our operational model. Getting into this game now, starting with a well-scoped MVP, positions Concepta to expand AI capabilities responsibly as the foundation matures.

💡
The Core Problem

Time, budget, and resource capacity are being lost to a reactive, lean operating model that was never designed to scale. We have been moving fast — but without the structure needed to deliver predictably. The result is ambiguity, rework, and client budgets spent on coordination overhead rather than meaningful outcomes. This initiative marks a deliberate shift: from reactive to predictable. To do this, we will need to revamp our operational model in partnership with both our internal team and our external account stakeholders — building a shared structure that gives clients clear visibility into where their budget is going, maximizes spend toward high-impact deliverables, and ensures work is delivered on time, on budget, and with quality as a baseline expectation — not an exception.

Operational Guidelines

How Requests Must Be Handled

These guidelines define the rules of engagement for the intake process. They apply to all team members and accounts once this framework is rolled out.

📥 Guideline 1 — All Requests Must Be Captured via a Standardized Client Request Board Mandatory

Every account must have a Client Request Tracking Board — this is where all incoming requests are captured, tracked, and progressed through the delivery lifecycle. The request board is the single source of truth for a client's active and pending requests at any given time.

Basecamp is the designated tool for the client request board across all accounts. The majority of our accounts already use Basecamp for client communications, project updates, and collaboration — making it the natural home for request tracking. However, the request board feature requires a Basecamp plan upgrade. All accounts must be on Basecamp 4 with the Request Board feature enabled before this framework can be fully operationalized.

⚠️
Action Required — Basecamp Upgrade

Only a subset of accounts currently have a request board established. All remaining accounts will need to be upgraded to Basecamp 4 and have a Request Board created and configured before intake can be standardized across the organization.

Step 1 — Client / Requestor Submits a Request Form

Before a request can be processed, the client or requestor must submit a request using the standardized Concepta intake form. Two forms are available directly on the account's Basecamp Request Board — the requestor selects the appropriate form for their submission type: New Request / Enhancement or Reported Production Issue. For reported issues, the requestor also selects the severity level (Sev 1–4) based on definitions agreed with client stakeholders. The required fields differ between the two forms, and the AI Agent reads these fields to validate completeness, confirm the request category, assign the correct billing epic, and route to the right workflow.

🤖 Guideline 2 — The AI Agent Is the First Responder on Every Request AI-Assisted

Once a client submits a request via the Basecamp Request Board, the AI Agent automatically moves the item into the Validate & Assign New Request column — this is where all agent activity takes place before any human on the delivery team is involved. The agent handles data validation, request categorisation, ticket creation, epic assignment, and team notification in sequence. The delivery team does not begin analysis until the AI Agent has completed its steps and a human has confirmed ticket accuracy.

The AI Agent performs the following steps in sequence:

  • Step 1 — Validate required fields and confirm request category: The requestor selects the appropriate intake form on the Request Board — either the New Request / Enhancement form or the Reported Issue form — and submits their request. For reported issues, the requestor also assigns the severity level (Sev 1–4) based on pre-agreed definitions confirmed with client stakeholders. Once the item is submitted, the AI Agent picks it up, scans all data values to confirm completeness, and categorises the request type based on what was submitted.
    • Reported Issue form submitted with severity level assigned → categorised as a Production Issue → routed to the production issue workflow
    • New Request / Enhancement form submitted → categorised as a New Request or Enhancement → routed to the new request workflow
    • If required fields are missing or insufficient, the AI Agent notifies the requestor of what's missing. The item remains held in the Validate & Assign column in a pending state until all required data is confirmed and resubmitted.
  • Notification 1 — Validate & Assign acknowledgement: As soon as the item is moved into the Validate & Assign New Request column, the AI Agent sends an automated notification to the requestor confirming receipt and informing them that their submission is being validated. The notification includes a link to the Basecamp Request Board so the requestor can track the status of their item at any time.
  • Step 2 — Route to the correct Basecamp column based on request type: Once the intake is validated and the category is confirmed, the agent moves the item out of the Validate & Assign New Request column and into the appropriate column on the Basecamp Request Board:
    • New Request / Enhancement → item is moved to Analyzing / Scoping / Estimating
    • Production Issue → item is moved to Triage / Troubleshooting
  • Step 3 — Create Zoho ticket, assign epic and backlog bucket based on request type: Based on the confirmed category, the AI Agent takes the following actions simultaneously — creating the Zoho ticket, assigning the correct epic, and routing to the appropriate backlog bucket:

  • Notification 2 — New Request status update: When the AI Agent moves a validated new request or enhancement into the Analyzing / Scoping / Estimating column, it sends an automated notification to the requestor confirming the item has been validated, assigned to the team, and is now in analysis. The notification includes the Basecamp Request Board link so the requestor can monitor progress.

  • Step 5 — Notify the account Slack channel: The agent sends a notification with the ticket link, request summary, requestor name, confirmed category, assigned epic, and Zoho Backlog Sprint Bucket so the broader team has immediate visibility.

The Form Selection Is the Routing Mechanism

Because production issues and new requests have separate intake forms with different required fields, the form the requestor selects tells the AI Agent the request type. The requestor also assigns the severity level on the Reported Issue form based on pre-agreed definitions. The AI Agent reads these values, validates completeness, and routes accordingly — the human oversight step becomes a spot-check rather than a categorization task.

Human oversight is required after Step 5 — the PDM reviews AI-created tickets for accuracy before the request advances into Stage 3 (internal triage).

🔄 Guideline 2b — The AI Agent Monitors Continuously After Handoff AI-Assisted

The AI Agent's role does not end at handoff. Once a request is routed to the appropriate team, the agent runs a lightweight continuous monitoring role throughout the workflow — removing coordination overhead from the PDM and ensuring nothing stalls without a nudge. All monitoring actions are rule-based triggers, not autonomous decisions.

For New Request / Enhancement workflows, the AI Agent monitors:

  • Ticket inactive beyond planning ceremony window → flags to PDM + SA in Slack
  • Scope deliverable moved to review column → automatically notifies requestor
  • No requestor response within SLA window → sends automated reminder to requestor
  • Revision submitted by requestor → notifies Analysis Team, moves card back, logs revision on ticket
  • Planning/Ops budget approaching defined threshold → flags to PDM before overage occurs
  • Scope approved by requestor → moves Basecamp item to "Approved / Prioritize," updates ticket, notifies Analysis Team

For Production Issue workflows, the AI Agent monitors:

  • Ticket inactive beyond SLA update cadence → flags to Prod Support team in Slack
  • Prod Support requests clarification from requestor → notifies requestor automatically, logs on ticket
  • No requestor response to clarification within SLA window → sends automated reminder
  • SLA breach threshold reached → re-escalates in Slack, flags to PDM
  • Budget overage approved by stakeholder → notifies Prod Support team they are clear to proceed
  • Prod Support budget approaching defined threshold → flags to PDM before overage occurs
  • Fix verified in production → automatically notifies requestor and prompts them to confirm resolution on the Request Board
💡
Lean by design — built by our internal labs team

All AI Agent actions are rule-based triggers: reading ticket state, moving cards, sending pre-configured notifications, and posting Slack messages. There is no complex AI reasoning involved. The implementation is owned and built by Concepta's internal labs team, who work across our tooling stack — Basecamp, Zoho, and Slack integrations are well within their wheelhouse. We are starting lean and will expand the agent's role as we build confidence through the first implementation.

🏷️ Guideline 3 — Every Request Must Be Assigned to the Correct Billing Epic Billing Critical

The AI Agent automatically assigns the billing epic at the time of ticket creation. However, the PDM must verify this assignment is correct during the human oversight review step. No time should be logged against a ticket that does not have a billing epic assigned.

Request Type Billing Epic / Service Bucket Notes

Enhancement to Existing Feature Planning / Processing / Ops Analysis, scoping, and estimation hours tracked here
Reported Production Issue Production Support Draws from Prod Support baseline; SLA response times apply

📊
Budget Hours Must Be Associated to Each Epic

Each billing epic should reflect the account's allocated baseline hours for that service bucket. This allows the team — and the client — to see remaining hours per bucket at any point in the billing cycle, enabling informed decisions about whether a new request can be absorbed or needs to be deferred or re-prioritized.

⚠️
Incorrect Epic Assignment = Budget Reporting Errors

Misassigned tickets corrupt service bucket tracking and make it impossible to accurately report burn rate, adjust baselines, or make informed capacity decisions with clients.

📅 Guideline 4 — Billing Boards Open and Close Per Account Cycle Billing Critical

Each client account operates on a predefined billing cycle. Most accounts run on either a 3-week or 4-week board cadence. Boards open and close in alignment with these cycles, and all time entries must fall within the correct active board.

  • At the start of each cycle, confirm the board is open and service bucket baselines are current.
  • Any bucket trending over its baseline should have buffer hours reassigned to it — and this data is used to recalibrate baselines at the next contract review.
  • Clients should be regularly updated on bucket utilization so they can make informed prioritization decisions.
🔁 Guideline 5 — Two Feedback Loops May Return a Request to the Requestor Process Rule

A request can be returned to the requestor at two distinct points in the workflow:

  • Loop 1 — AI Agent: Triggered when required intake fields are missing or insufficient. The AI Agent flags the gap and the PDM coordinates with the requestor to provide additional information before the ticket is created.
  • Loop 2 — Internal Team: Triggered during the Internal Review stage when the PDM or SA determines the request is unclear, ambiguous, or requires additional context to properly evaluate. The team documents the specific questions and informs the client.

In both cases, the request is placed in a pending / on-hold state and does not advance until the requestor responds and the missing information is confirmed complete.

📊 Guideline 6 — Service Bucket Baselines Must Be Defined Per Account Operational

Each account must have a defined set of service bucket baselines — negotiated with client stakeholders — before this framework can be fully operationalized. These baselines determine how hours are allocated, tracked, and reported each billing cycle.

  • Reference model (Service Finance, 4-week cycle): Production Support baseline was defined at 45 hrs/cycle with a dedicated Dev and QA resource assigned to specific office hours for monitoring and triage. SLA response times were defined per severity level.
  • Baselines should be reviewed regularly with client stakeholders and adjusted based on observed usage patterns and shifting business priorities.
  • New Requests (enhancements and new functionality) are baked into the Planning/Ops service bucket — not a separate line item — reflecting the time required to analyze, estimate, and route the request.
📝 Guideline 7 — Documentation Is Not Overhead. It Is the Infrastructure. Critical

This is the single biggest gap standing between where Concepta is today and where this initiative needs to get to. AI Agents cannot operate on informal knowledge, undocumented processes, or assumptions that live in someone's head. They require clearly defined, explicitly documented workflows and account-specific content models in order to make correct decisions, route accurately, and notify the right people at the right time.

Documentation must exist at two levels before the AI Agent can be enabled on any account:

  • Workflow documentation — every step, every decision point, every trigger, every rule must be explicitly written down. The diagrams in this initiative are the start — but they must be accompanied by the operational detail that makes each step executable. Ambiguity is the enemy of automation.
  • Account-specific content models — each account requires its own configuration: severity definitions, SLA windows, service bucket baselines, escalation contacts, team assignments, ceremony schedules, budget thresholds, and notification templates. The AI Agent reads from these. If they are incomplete, outdated, or missing — the agent makes wrong decisions.
⚠️
This applies to the entire team — not just the AI Agent configuration

The discipline of documentation must be adopted as a core operating practice. Every process that runs through this framework — intake, triage, analysis, production support, budget tracking — depends on shared, written, up-to-date documentation. Without it, the framework reverts to informal coordination and the AI Agent cannot add value. Documentation is how we scale.

Production Issue workflow — additional steps in Stage 3

Production issues follow a dedicated workflow with additional checkpoints before a fix is implemented: budget check against remaining Prod Support allocation (with deployment budget and buffer as fallback); stakeholder escalation and sign-off if overage is required; RCA initiation for issues meeting the pre-agreed business impact criteria (runs alongside the fix, not as a gate); release version alignment (hotfix vs. scheduled release per severity and client agreement); fix implementation and QA validation in staging; production deployment; and requestor confirmation of resolution. Next Steps

Seven Actions. In This Order.

These are the actions required to move from framework to live operation. Each step enables the next — the sequence matters.

00

Working session cadence — established ✅

Tuesdays 4–5pm. Live working sessions focused on resource allocation and priorities — not written updates. This is the operating rhythm for everything below. Already confirmed between Raul and Anne.

01

Redefine account structure with each client

Re-align on service buckets, budget baselines, SLAs, severity definitions, schedules, and communication channels per account. Start with Service Finance as the reference model and work through all active accounts. This step unlocks everything else — nothing can be configured, tooled, or automated until account-level alignment is confirmed with client stakeholders.

02

Define and build the delivery toolkit

Configure the core delivery templates that support this framework per account — scope document template, Smartsheet Estimator (covering all activities required to achieve DoD), Smartsheet Project Plan template (initiated at start of analysis), and RCA documentation template. All templates must be validated per account before any work moves through the process.

03

Align on planning and ops ceremonies

Define the full ceremony schedule per account — cadence, attendees, and committed resources. Fold new request processing into those ceremonies so intake analysis happens within the established planning rhythm. Lock resource allocation across accounts to prevent overlap and double-booking of shared resources.

04

Pilot on the corporate site

Use the corporate site project as a controlled pilot — the PDM team acts as both the client (submitting requests) and the delivery team (processing them). Run the full intake workflow end to end: submit a request, validate it, create the ticket, move the card, route it to the correct workflow, process it through analysis or triage. Find every gap, validate the model, and prove it works before it touches a live client account. This is the most important step before broader rollout.
