Intake & Request Tracking

The two client-facing boards that work together across the request lifecycle — Basecamp for intake and per-request communication, Smartsheet for the full tracker. This page covers the forms, the fields, the boards, and how work moves from submission through release.

🚶
Phase 1 — Manual Pilot: PDM and DM perform all AI Agent steps This document describes the intended process including future AI Agent automation. During the manual pilot, every step attributed to the AI Agent is performed manually — by the PDM for intake validation and card management, and by the DM at the Validate & Assign operational gate (team assignment, budget confirmation, Smartsheet project plan initiation). The AI Agent is introduced in Phase 2 — after the manual process is validated and the team has signed off.
📋
Forms are built in Smartsheet — one set per account Each client account has its own New Request and Reported Issue forms created directly on their Client Request Tracker sheet. Both forms submit into the portfolio automatically with Status = Not Started and Request Type pre-populated. During the manual pilot, the PDM submits on behalf of the requestor.

Board 1 · Entry & Communication
Basecamp Submission Board

The entry point for all requests and the client-facing communication surface. Clients click the Smartsheet form links from the board to submit. The PDM creates the corresponding card to track each request visually as it moves through the lifecycle, and the team comments back and forth with the client directly on the card.

Purpose
  • Submit new requests and reported issues via Smartsheet form links
  • Track each request visually as cards move through the 8-column lifecycle
  • Comment back and forth with the client on each request card
  • Client-facing real-time view of where every request stands

Board 2 · Full Request Tracking
Smartsheet Client Request Tracker

The structured tracker for the account. Every request lives here as a row with full lifecycle status, prioritization, release train sequencing, and the supporting artifacts attached — project plan, estimator, scope document, release plan. This is where the team and client see the complete picture.

Purpose
  • Full tracking visibility — all requests in one structured view
  • Track status from submission through release
  • Access project plans, estimators, scope docs, release plans inline
  • Used for prioritization and release train sequencing

💡
The boards are complementary — not duplicates

Basecamp is where the conversation happens — requests are submitted, discussed, and visually tracked card by card as the client sees progress in real time. Smartsheet is where the full picture lives — every request as a structured row with supporting artifacts, status detail, and release train sequencing. During the pilot, the PDM creates the Basecamp card manually after the Smartsheet form submission, keeping both surfaces in sync.

New Request / Enhancement Form

Used when a requestor wants to introduce new functionality, enhance an existing feature, or request work that is not an active production issue.

Routes to: New Request workflow · Analyze/Scope Request column · Planning/Ops/New Requests epic

Required fields: 5 (see below)

Reported Issue Form

Used when a requestor is reporting an active defect, incident, or production-impacting problem that requires triage and resolution.

Routes to: Production Issue workflow · Triage/Troubleshoot column · Production Support epic

Required fields: 5 (see below) — plus an optional attachment field for screenshots, artifacts, or videos

Basecamp Submission Board — reference structure. The top swimlane is the entry portal: two "click and copy" template cards that link to the Smartsheet New Request and Reported Issue forms. The bottom "Processing" swimlane holds the 8-column lifecycle — cards move manually through Submitted new item → Validate & Assign (DM) → Analyze/Scope Request OR Triage/Troubleshoot → Review Scope / Approve → Needs Prioritization → In Progress → Archive as work advances.

Every request lands as a row in the NEW REQUESTS grouping after form submission, then moves up through SCOPE/ANALYSIS → SCOPE APPROVED (Prioritize & Assign to Dev) → DEV & DEPLOYMENT → RELEASED as work advances (with BACKLOG for deferred items). Phase 1 Required Fields

5 Required Fields — Shared Across Both Forms

The New Request and Reported Issue forms share the same 5 required fields. Reported Issue adds one optional attachment field (see Section 2).

💡
Phase 1 forms are intentionally minimal — by team design

The team requested a minimal field set for Phase 1 to reduce client friction and speed up adoption. Fields may expand in later phases as the process matures and data gaps become clear. Today, the 5 fields below are what every submission must capture.

🔒
Two hidden fields auto-populate on submission

Request Type (New Request or Reported Issue) is pre-populated based on which form the requestor filled out — this is the routing mechanism. Status auto-populates to "Not Started" so every submission lands in the NEW REQUESTS grouping on the Smartsheet Request Tracker ready for the DM's operational gate.

1
Request Date
The date the request is submitted. Used for SLA clock start on reported issues and for tracking request age on new requests.
2
Request Title
A short, clear summary of the request or issue. Used as the row title on the Smartsheet Request Tracker and the card title on the Basecamp Submission Board.
New Request: "Export capability for dealer reporting dashboard" · Reported Issue: "Dealer app crashing on iOS 17 for iPad users"
3
Requestor
The name of the person submitting the request. Used for communication on the Basecamp card throughout the lifecycle and for the SLA response notification on reported issues.
4
Severity / Priority (Sev 1–4)
Selected from the pre-agreed definitions for this account. Used differently depending on form type — see the Severity Guidance section below for how each form type applies this value.
Sev 1: Critical · Sev 2: High · Sev 3: Medium · Sev 4: Low
5
Notes ("What is being requested and why?")
A free-text field where the requestor describes what they need and the underlying reason. For new requests, focus on the problem or goal rather than a prescribed solution. For reported issues, describe the issue, steps to reproduce if known, and impact.
New Request: "Dealers cannot export monthly reporting data needed for compliance reviews. Need an export button on the dashboard to pull data in CSV format." · Reported Issue: "Dealer app crashes when tapping Submit on application form. Happens on iOS 17, iPad only. 12 dealers reporting since 2/9 release."
2
Reported Issue — Additional Field

Supportive Screenshots / Artifacts / Videos

Optional — only on the Reported Issue form

+
Supportive Screenshots / Artifacts / Videos (Optional)
A file upload field where the requestor can attach screenshots, screen recordings, error messages, console logs, or any artifact that helps the production support team reproduce and diagnose the issue. Significantly accelerates triage when provided.
3
Severity Guidance

How Severity Is Used on Each Form

Same dropdown, different downstream use — severity is collected on both forms and drives different team actions depending on request type.

Reported Issue → SLA Response Time

Severity drives the SLA response time communicated to the requestor. Sev 1 (Critical) triggers immediate response; Sev 4 (Low) follows the standard response cadence for that account. The DM sends the SLA notification via the Basecamp card comment thread at the oversight step.

New Request → Prioritization Urgency

Severity informs prioritization urgency on the release train. It does not create an SLA commitment, but helps the team and requestor sequence work appropriately — a Sev 1 new request signals high business urgency and should be weighted accordingly during prioritization discussions on the Smartsheet Request Tracker.

⚠️
Severity is assigned by the requestor at submission

Severity definitions (Sev 1–4) are agreed with client stakeholders per account before go-live and are available to the requestor directly on the intake form. If the team determines the severity is incorrect after review, they reassign it and the DM updates the Basecamp card and any prior SLA notification accordingly.

What Happens After Submission

The DM's Operational Gate at Validate & Assign (DM)

Once submitted, the row lands on the Smartsheet Request Tracker and the PDM creates the Basecamp card in the Submitted new item column. Here's what happens next on each path during the pilot phase.

🔵
New Request Path — DM Operational Gate

The submission lands at the Validate & Assign (DM) column on the Basecamp Submission Board where the DM performs the operational gate: assigns the request to the appropriate analysis team based on the planning ceremony schedule, confirms budget availability in the Planning/Ops/New Requests epic, initiates the Smartsheet project plan with the analysis activities, and informs the requestor on the Basecamp card comment thread when analysis will start. The DM collaborates with the PDM if unsure who should analyze the request during the planning call. In Phase 2, the AI Agent reads the submission and surfaces budget data to the DM — the operational gate remains a human decision.

🔴
Reported Issue Path — DM Operational Gate

The submission lands at the Validate & Assign (DM) column with severity already assigned by the requestor. The DM performs the operational gate: sends the SLA response notification matching the submitted severity via the Basecamp card comment thread, assigns the issue to the production support team based on the production scheduler, confirms budget availability in the Production Support epic, initiates the Smartsheet project plan with the triage/troubleshoot activities, and informs the requestor on the Basecamp card when triage will begin per SLA. The DM collaborates with the PDM if severity or resourcing needs clarification. In Phase 2, the AI Agent fires the SLA notification automatically and surfaces budget data to the DM — the operational gate remains a human decision.

🔮
Future Consideration — Smartsheet Consolidation (Anne's view · for team consideration)

As clients become increasingly comfortable with Smartsheet through their exposure to the Request Tracker, we should consider consolidating status tracking and per-request communication onto Smartsheet as well — eliminating the parallel Basecamp board entirely. Forms already live on Smartsheet; the next step would be bringing the visual status and client comms there too, giving clients a single board surface for both submitting and tracking requests. This is a directional idea — not yet agreed with the team. To be discussed and pressure-tested as adoption matures.
