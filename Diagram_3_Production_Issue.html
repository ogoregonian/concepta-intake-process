AI Agent
📌 Prerequisites
  • Zoho Sprints: Backlog Sprint bucket for Reported Issues configured · Bug ticket template defined
  • Slack channel active with Production Support team tagged as assignees
  • Severity level definitions (Sev 1–4) agreed and documented by client stakeholders — includes response times, communication channels, escalation paths, and office hours per severity
  • All severity + SLA definitions loaded into the intake form and Bug ticket template so they are captured at submission and available on the ticket for the AI Agent to read
  • SLA update cadence defined per severity — used to detect ticket inactivity and trigger Slack nudges
  • Budget threshold alert level defined — percentage of Prod Support budget that triggers a PDM flag
  • Notification templates configured per severity and trigger type (clarification request, SLA breach, fix verified, etc.)
  • Basecamp Request Board: "Triage / Troubleshooting" and "Done / Closed" columns active
🤖 AI Agent scope in this lane: Handles the initial handoff to the Production Support team — reads severity and all associated details from the ticket, fires the corresponding pre-defined notification, and tags the assignee in Slack. Then steps into a continuous monitoring role throughout triage and resolution — nudging stalled steps, relaying clarification requests, flagging SLA breaches and budget thresholds, closing the loop on stakeholder approvals, and notifying the requestor automatically when the fix is live.

Production Support Team
📌 Prerequisites
  • Dedicated production support team assigned (Dev + QA)
  • Production support office hours defined and communicated to client
  • Account SLA definitions and severity response times agreed with client stakeholders
  • Account budget structure defined — production support bucket, deployment budget, and buffer allocation established
  • Budget overage escalation path defined — stakeholder contacts identified per account
  • Release versioning approach agreed — defines how and when fixes are deployed (hotfix vs. scheduled release)
  • RCA criteria defined per account — specifies which severity levels and business impact thresholds require a Root Cause Analysis document
  • Zoho Sprints access to manage Bug ticket lifecycle, activity logging, and sprint assignment
  • Basecamp Request Board access to manage column movement
  • Escalation path defined for Sev 1 issues (e.g. on-call protocol)

Severity Level Reference Defined and agreed by client stakeholders before go-live — the AI Agent reads severity from the ticket and triggers the corresponding pre-configured response
Level Description Example SLA Response Target Outside Office Hours? Escalation Protocol
SEV 1 Critical — production system fully down or data loss occurring. All users impacted. Application unreachable, data corruption, complete service outage [per account SLA] Yes — on-call applies Immediate on-call escalation · Stakeholder notification required
SEV 2 High — major feature broken or significantly degraded. Large subset of users impacted. Core workflow blocked, critical API failing, major performance degradation [per account SLA] Yes — SLA window governs Response within SLA window · Stakeholder informed
SEV 3 Medium — functionality impaired but workaround exists. Limited user impact. Non-critical feature broken, UI bug affecting subset of users, minor data display issue [per account SLA] No — office hours only Reviewed during next available support window
SEV 4 Low — cosmetic issue or minor inconvenience. No business impact. Typo in UI, minor styling issue, low-priority enhancement flagged as a bug [per account SLA] No — office hours only Reviewed in next available sprint cycle
* All severity definitions, SLA response targets, communication channels, escalation paths, and office hours are defined and agreed by the client and internal stakeholders before go-live. This table is the structural template — values are populated per account during onboarding.
